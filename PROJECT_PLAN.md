## 一、产品定位与核心场景

### 1.1 工具用途

商家通过采集软件获取竞品/爆品的产品数据，导入本工具后：
- **AI仿写文案**：微调标题、卖点、描述等文案字段，保留产品核心特征但避免直接抄袭
- **AI重新生成图片**：主图+副图全部通过AI生图模型重新生成，避免图片侵权
- **校验完整性**：按亚马逊SP-API规则自动检查漏填项、格式错误
- **批量上架**：通过SP-API putListingsItem接口批量提交，支持多账号切换

**一句话总结**：竞品数据采集 -> AI铺货改造 -> 一键批量上架

### 1.2 用户工作流

```
采集软件导出Excel
    |
导入本工具（自动识别列名 + 变体关系）
    |
AI仿写文案（标题/卖点/描述/搜索词）<- 原始 vs AI 对比确认
    |
AI重生图片（主图 + 副图1~8）<- 原图 vs AI图 对比确认
    |
自动校验（必填项 + 长度 + 变体完整性 + 合规字段）
    |
选择上架账号 -> SP-API批量提交
    |
获取ASIN -> 完成铺货
```

## 二、SP-API标准数据模板

### 2.1 字段分类与AI处理判断

**核心原则**：只有"文案类"和"图片类"字段需要AI处理，其他字段保持采集原始值或手动填写。

#### AI需要处理的字段（仿写/重生）

| 字段 | AI处理类型 | 说明 |
|------|-----------|------|
| `item_name` | **仿写** | 基于竞品标题微调措辞，保留产品核心关键词 |
| `bullet_point_1` | **仿写** | 基于竞���卖点改写，突出差异化 |
| `bullet_point_2` | **仿写** | 同上 |
| `bullet_point_3` | **仿写** | 同上 |
| `bullet_point_4` | **仿写** | 同上 |
| `bullet_point_5` | **仿写** | 同上 |
| `description` | **仿写** | 基于竞品描述改写，保持产品特征 |
| `search_terms` | **AI生成** | 基于标题+卖点自动提取/优化搜索关键词 |
| `main_image_url` | **AI重生** | 通过AI生图模型重新生成主图 |
| `image_2` ~ `image_9` | **AI重生** | 副图也通过AI生图模型重新生成 |

#### 不需要AI处理的字段（保持原始/手动填写）

| 字段组 | 字段 | 原因 |
|--------|------|------|
| **产品身份** | `SKU` | 卖家自定义编号 |
| | `product_type` | 亚马逊标准分类值，不可改 |
| | `brand` | 品牌名称，不可随意改 |
| | `manufacturer` | 制造商名，不可改 |
| | `model_number` | 型号，不可改 |
| | `UPC`/`EAN`/`GTIN` | 条码标识，不可改 |
| **报价** | `price`, `currency` | 卖家自定定价 |
| | `quantity` | 库存数量 |
| | `condition_type` | 商品状况（new等） |
| | `fulfillment_channel` | 配送方式（FBA/FBM） |
| **物流** | `item_weight`, 尺寸字段 | 物理参数，不可改 |
| | `package_*` | 包装参数，不可改 |
| **属性** | `color`, `size`, `material` | 变体属性值，不可改 |
| | `department`, `target_gender` | 分类属性 |
| | `age_range`, `country_of_origin` | 产品属性 |
| **合规** | `batteries_required` | 是否需要电池（Yes/No） |
| | `batteries_included` | 是否含电池 |
| | `battery_type` | 电池类型 |
| | `hazmat_declaration` | 危化品申报 |
| **变体** | `parent_sku` | 父商品SKU |
| | `parentage_level` | parent/child |
| | `variation_theme` | 变体主题（Color/Size等） |

## 三、前端UI架构（类Excel电子表格风格）

> **设计理念**：参考店小秘、妙手等跨境电商ERP工具的UI风格，采用高信息密度的电子表格布局。
> 铺货场景数据量大，UI必须像Excel一样清晰紧凑，方便快速浏览和操作大批量商品数据。

### 3.1 设计原则

| 原则 | 说明 |
|------|------|
| **表格优先** | 主界面就是一张大表格，所有商品数据一目了然，而非卡片/仪表板风格 |
| **高信息密度** | 行高紧凑(48-64px)，13-14px字体，单屏可见更多数据 |
| **右侧抽屉编辑** | 点击商品行 -> 右侧滑出编辑抽屉(500px)，不离开列表页 |
| **筛选栏+状态Tab** | 表格上方设快捷状态Tab + 搜索/筛选栏，快速定位商品 |
| **选中后批量栏** | 勾选商品后底部浮出批量操作栏，未勾选时隐藏 |
| **列可配置** | 用户可自选显示哪些列，适应不同工作场景 |

```
7. 批量处理中 -> 进度浮层显示 [====--] 28/42 -> 可取消
8. 校验 -> 表格校验列实时更新 V/X -> 点击X查看具体错误
9. 提交上架 -> 选择账号 -> 预览模式验证 -> 正式提交 -> 获取ASIN
```

### 4.3 数据流

```
                          +---------------------+
                          |  采集软件导出Excel    |
                          +----------+----------+
                                     |
+------------------------------------------------------------+
| Step 1: 导入                                                |
|  POST /api/upload -> read_input() -> detect_columns()      |

                              |
+------------------------------------------------------------+

|  -> 写入AI列(ai_title, ai_bullet_1~5, ai_desc, ai_kw)     |
|  -> 前端对比展示: 原始 vs AI                                 |
+-----------------------------+------------------------------+
                              |
+------------------------------------------------------------+
| Step 3: AI重生图片                                          |
|  单条: POST /api/process-single {action:"image"}            |
|  批量: POST /api/process-images                             |
|  -> 主图+副图全部AI重新生成                                  |
|  -> 前端对比展示: 原图 vs AI图                               |
+-----------------------------+------------------------------+
                              |
+------------------------------------------------------------+
| Step 4: 校验                                                |
|  POST /api/validate {skus:[...]}                            |
|  -> 必填项检查: SKU, title, price, UPC                      |
|  -> 长度检查: title<=200, bullet<=500, search<=250bytes     |
|  -> 变体检查: parent有child、child有parent_sku              |

+------------------------------------------------------------+
```

## 五、批量处理设计

### 5.1 规模控制策略

| 参数 | 默认值 | 可配置 | 说明 |
|------|--------|--------|------|
| **单次导入上限** | 500行 | 是（设置页） | Excel最大行数，超出提示分批 |
| **批量AI文案并发** | 3 | 是（设置页） | 同时发送AI请求数 |
| **批量AI生图并发** | 2 | 是 | 生图比文案更耗时，降低并发 |
| **SP-API上架间隔** | 300ms | 否 | 亚马逊API限流 |
| **单批上架上限** | 50 | 否 | 每批提交数量 |

### 5.2 批量处理流程

```
用户勾选N条商品 -> 点击"批量AI文案"
    |
前端显示进度弹窗: [========--] 28/42  当前: TEE-RED-S
    |
后端线程池处理:
  - 队列: 按顺序排入任务
  - 并发: 最多3个同时处理
  - 进度: 每完成一条更新 /api/status
  - 错误: 单条失败不影响整体，记录到issues字段
  - 可中断: 前端可发送取消请求
    |
完成后: 刷新表格，Toast通知结果
```

### 5.3 大批量优化

- **超过100条**：提示"将分批处理，预计耗时约X分钟"
- **超过300条**：建议分多个Excel文件导入
- **进度持久化**：处理进度写入Excel，中断后重新导入可从断点继续（检测ai_status列）

## 七、关键设计决策记录

| # | 决策 | 选择 | 原因 |
|---|------|------|------|
| 1 | 页面数量 | 2页（工作台+设置） | 所有AI操作在表格行内完成，无需单独页面 |
| 2 | AI处理方式 | 仿写（非翻译） | 基于竞品文案微调，保留产品特征，避免抄袭 |
| 3 | 图片处理范围 | 主图+副图全部 | 避免图片侵权，全部AI重新生成 |
| 4 | 合规字段 | 包含 | 电池/危化品等合规字段也需要在模板中 |
| 5 | 多账号 | 支持 | 上架时可选择不同账号/站点 |
| 6 | 批量规模 | 可配置(默认500) | 用户可在设置中调整上限 |
| 7 | 详情编辑方式 | 右侧抽屉(5个Tab) | 参考店小秘/妙手：不离开列表页，抽屉内编辑详情 |
| 8 | Excel模板 | 静态通用+动态Schema | 两种模板获取方式并存 |
| 9 | UI风格 | 类Excel高密度表格 | 参考店小秘/妙手ERP，Ant Design配色，适合大批量数据浏览 |
| 10 | 表格列 | 可配置显示/隐藏 | 不同工作阶段关注不同字段，列配置提高灵活性 |
