## 二、功能缺陷（影响业务流程）

### 2.1 图片生成API调用方式与当前模型不匹配

**文件**: `stage1_pipeline.py:440-516`

**现状**: 使用 `chat.completions.create` 发送图片，期望AI在文本响应中返回base64图片数据。

**问题**:
- 当前配置的 `AI_IMAGE_MODEL=gemini-3-pro-image-preview` 不是通过chat返回base64 inline的
- 代码用正则 `re.search(r'data:image/\w+;base64,...')` 从文本响应中提取图片，这种方式不可靠
- OpenAI的图片生成应该用 `client.images.edit()` 或 `client.images.generate()`

**修复方案**: 根据实际使用的AI供应商（Gemini/OpenAI/第三方），适配对应的图片生成API。

---

### 2.2 Submit接口不实际执行提交

**文件**: `web/app.py:1446-1463`

**现状**: 正式提交模式只构建了 `body_preview` 返回，没有调用 `listings.put_listings_item()`。

```python
# 正式提交模式 — 但其实只是构建了body就返回了
body = mapper.build_put_body(product)
results.append({
    'sku': sku,
    'status': 'PENDING',
    'message': '已加入提交队列',
    'body_preview': body,    # 没有实际提交!
})
```

**修复**: 需要添加实际的SP-API调用逻辑。

---

### 2.3 图片处理只处理主图，忽略副图

**文件**: `stage1_pipeline.py:363-421`

**现状**: `_process_images` 只找到第一个有效图片URL就处理并返回，不处理 image_2~image_9。

**PROJECT_PLAN要求**: 主图 + 副图1~8 全部通过AI重新生成。

---

### 2.4 批量处理无并发，纯串行

**文件**: `stage1_pipeline.py:96`

**现状**: `for idx, item in enumerate(data)` 逐条串行处理。

**PROJECT_PLAN要求**: 文案并发3、生图并发2。

**修复方案**: 使用 `concurrent.futures.ThreadPoolExecutor` 或 `asyncio` 实现并发。

---

### 2.5 `.env` 配置更新后不重载

**文件**: `web/app.py:753`, `config.py:39-40`

**现状**: Web页面修改API Key后保存到 `.env`，但运行中的进程仍然用旧的 `Config` 对象。返回消息写的是"重启生效"，用户体验差。

**修复方案**: `get_config()` 改为重新读取 `.env`，或在保存配置后调用 `load_dotenv(override=True)` 并刷新 `Config` 实例。

## 五、你遇到的400错误分析

```
Error: API Error: 400 {"error":{"message":"请求参数不合法，请检查后重试"}}
```

**当前配置**:
```env
AI_API_BASE=https://api.kk666.online/v1
AI_TEXT_MODEL=gemini-3-flash-preview
AI_IMAGE_MODEL=gemini-3-pro-image-preview
```

**可能原因（按概率排序）**:
1. **模型名称不被代理支持** — `gemini-3-flash-preview` 可能已更名或该代理不提供此模型
2. **Prompt过长** — 产品信息 + prompt模板拼接后超出该代理的输入限制
3. **参数格式差异** — 第三方代理对 `max_tokens`、`temperature` 等参数的允许范围与OpenAI不同
4. **图片base64过大** — 图片编辑请求中base64数据超出请求体限制

**排查建议**: 在代理商文档中确认可用模型列表，或用 `curl` 单独测试一次简短请求确认模型是否可用。
