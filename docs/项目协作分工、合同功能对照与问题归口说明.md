## 4. 合同约定功能：逐条对照、是否完成、完成得怎么样

本节严格对照合同附件原文，逐条说明每项约定功能的交付情况。此部分为开发者 2 的合同义务，承担正常交付责任。

### 4.1 第一阶段：商品内容技术处理（合同附件第一节，费用 2,000 元）

合同附件第一节共列明 4 项内容，逐项对照如下：

**① 输入：甲方提供的 Excel 模板数据 + 图片素材（链接或文件）**

- 交付状态：**已交付 ✓**
- 系统支持导入 `.xlsx`（新版 Excel）和 `.xls`（旧版 Excel）两种格式。导入后系统通过一套包含 100+ 组中英文列名别名的映射表（如"商品名称"/"item_name"/"标题"均可识别为标题列），自动将 Excel 列名映射到内部标准字段，商家和采集侧无需严格对齐固定列名。
- 图片素材方面，系统支持公网 URL 链接（如 `https://` 开头的图片直链）和本地文件路径两种方式。导入后图片会在工作台表格中显示缩略图预览。
- 导入过程包含文件格式校验、MIME 类型验证、文件大小限制（50MB）等安全检查。
- 对应代码：`core/excel/processor.py`（ExcelProcessor.read_input、COLUMN_ALIASES）、`web/app.py`（/api/upload）

**② 图片处理：图片重生成（如背景替换）（按甲方确认风格/规则）**

- 交付状态：**已交付 ✓**
- 系统通过 AI 图片编辑接口（支持 OpenAI Images Edits 协议和 Gemini generateContent 协议两种 AI 后端）对商品图片进行背景替换处理。处理范围包括主图（main image）和最多 8 张副图（other images），共计可处理 9 张图片。
- 支持两种操作模式：
  - **单条处理** — 在抽屉图片 Tab 中针对单个 SKU 的单张图片进行换背景
  - **批量处理** — 勾选多个 SKU 后一键批量处理，支持只处理主图或全部图片
- 生成后的图片保存为本地 JPEG 文件（按 SKU + 唯一标识命名，避免覆盖），如配置了 S3 媒体存储还可自动上传获取公网地址。
- AI 生成过程内置重试机制（可配置重试次数），网络超时或 AI 服务临时不可用时会自动重试。
- 对应代码：`core/ai_client.py`（ai_image_edit、ai_image_edit_url、_ai_image_edit）、`stage1_pipeline.py`（_process_images、_ai_image_edit_url_with_retry）

**③ 文案处理：标题/描述等字段生成或改写（按甲方确认规则）**

- 交付状态：**已交付 ✓**
- 系统通过 AI 文本生成接口（同样支持 OpenAI chat.completions 协议和 Gemini generateContent 协议）对以下 4 类核心字段进行生成或改写：
  - **标题（item_name）** — 生成后自动截断至 200 字符内（Amazon 限制）
  - **五点描述（bullet_point_1~5）** — 每条自动截断至 500 字符内，共生成 5 条
  - **商品描述（product_description）** — 自动截断至 2,000 字符内
  - **后台搜索关键词（generic_keyword）** — 自动按站点字节限制裁切（美国站 250 字节）
- 支持两种操作模式：
  - **单条处理** — 在抽屉文案 Tab 中针对单个字段点击"重新生成"
  - **批量处理** — 勾选多个 SKU 后一键批量生成，支持选择生成范围（全部 / 只标题 / 只五点 / 只描述 / 只搜索词）
- AI 生成时自动注入 Amazon 专家角色 system prompt，提升输出格式稳定性。生成过程支持取消操作。
- 每个 prompt 模板均包含 `{product_type}`（产品类型）和 `{language}`（目标站点语言）参数，可根据商品类目和目标市场自动适配。
- 对应代码：`core/ai_client.py`（ai_text、AMAZON_SYSTEM_PROMPT）、`stage1_pipeline.py`（_generate_text_v2）、`core/prompts/amazon_prompts.py`（TITLE_PROMPT、BULLET_POINTS_PROMPT、DESCRIPTION_PROMPT、SEARCH_TERMS_PROMPT 等 13 个 prompt 模板）

**④ 输出：处理完成的 Excel 文件（字段与模板一致）**

- 交付状态：**已交付 ✓**
- AI 处理结果实时写回原始 Excel 文件。写回方式为：原始值保留在原列不动，AI 结果写入新的"AI 标题""AI 卖点 1~5""AI 商品描述""AI 搜索关键词""AI 主图路径"等列，便于商家做前后对比审阅。
- 支持两种格式导出：
  - **对比格式** — 原始值与 AI 值并排显示，适合审阅
  - **SP-API 格式** — 字段名与 Amazon SP-API 接口一致，可直接用于第二阶段提交
- Excel 写回操作支持 `.xlsx` 和 `.xls` 两种格式，写入时保留原始文件的样式和格式（.xls 格式通过 xlrd/xlwt 兼容处理）。
- 对应代码：`core/excel/processor.py`（write_comparison_output、write_output）、`web/app.py`（/api/export）

**结论：第一阶段合同约定 4 项功能已全部交付。**

### 4.2 第二阶段：自动化提交流程（合同附件第二节，费用 2,000 元）

合同附件第二节共列明 3 项内容，逐项对照如下：

**① 将输出 Excel 映射为平台接口请求并提交**

- 交付状态：**已交付 ✓**
- 系统读取第一阶段输出的 Excel 文件，自动将每行商品数据映射为符合 Amazon SP-API `putListingsItem` 接口要求的 attributes JSON 格式。映射逻辑覆盖 Amazon 全部 6 大字段分组：
  - **Product Identity（产品身份）** — 标题、品牌、UPC/EAN/GTIN、制造商、型号等
  - **Product Details（商品详情）** — 描述、五点、搜索词、颜色、尺寸、材质等
  - **Images（图片）** — 主图 + 最多 8 张副图的 media locator
  - **Offer（销售条款）** — 价格、币种、库存、配送方式、商品状态等
  - **Shipping（物流）** — 商品尺寸、重量、包装尺寸等
  - **Safety & Compliance（合规）** — 原产国、电池信息、危险品声明等
- 此外还支持变体（Variation）字段映射，包括 parentage_level（父/子体）、parent_sku、variation_theme、child_parent_sku_relationship 等。
- 提交方式为通过 Amazon SP-API `putListingsItem` 接口逐条提交至美国站（marketplace_id: ATVPDKIKX0DER）。提交前自动通过 LWA OAuth2 获取 access_token（线程安全，支持自动刷新）。
- 对应代码：`amazon/mapper.py`（FieldMapper.build_listing_attributes、build_put_body，共 1,289 行映射逻辑）、`amazon/listings.py`（ListingsAPI.put_listings_item）、`amazon/auth.py`（AmazonAuth，线程安全 token 管理）、`stage2_pipeline.py`（Stage2Pipeline.run）

**② 提交成功获得平台"已接收/处理中/草稿"等回执或状态**

- 交付状态：**已交付 ✓**
- 提交后系统接收并解析 Amazon 返回的完整响应，处理以下状态：
  - `ACCEPTED` — Amazon 已接受提交请求，进入异步处理
  - `ACCEPTED_WITH_WARNINGS` — 已接受但有警告信息
  - `INVALID` — 数据验证未通过，附带具体 issues 列表
  - `ERROR` — 接口调用失败，附带错误原因
- 每个 issue 包含：错误码（code）、严重级别（severity）、涉及字段名（attributeNames）、错误描述（message）。系统会为常见错误码附加中文说明和修复建议（如"UPC 条码无效"→"请核实 UPC 是否为有效的 12 位数字"）。
- 如果 Amazon 在响应中返回了 ASIN，系统会自动提取并回填。如未返回，系统会在提交 ACCEPTED 后自动补查一次 Listings API 尝试获取 ASIN。
- Token 过期时自动刷新并重试一次，无需人工干预。
- 对应代码：`amazon/listings.py`（_normalize_put_response、resolve_submission_asin、_ISSUE_HINTS 错误码映射表）

**③ 输出：提交日志/结果汇总（含成功/失败原因提示，失败原因以平台回执为准）**

- 交付状态：**已交付 ✓**
- 提交结果以两种形式输出：
  - **Excel 回写** — 逐 SKU 将提交状态、提交 ID、ASIN、错误详情等字段写回原始 Excel 文件，商家重新打开文件即可看到每行的提交结果
  - **JSON 日志** — 输出 JSON 格式的完整提交日志文件，包含提交时间、统计摘要（总数/成功/失败/跳过）、以及每个 SKU 的完整 issues 列表
- 失败原因直接展示 Amazon 平台的原始回执信息（code + message），确保"失败原因以平台回执为准"这一合同要求得到落实。
- 对应代码：`stage2_pipeline.py`（_generate_report）、`web/app.py`（/api/submission-history、任务中心相关接口）

**结论：第二阶段合同约定 3 项功能已全部交付。**

## 5. 超额完成：合同功能基础上主动增强的部分

以下内容的**基础功能属于合同约定**（已在第 4 节确认交付），但开发者 2 在实现过程中做到了显著超出合同基础描述的程度。超额部分并非合同要求，而是为了让商家实际使用时效果更好、出错更少而主动增强的。

### 5.1 图片处理方面的超额

合同只要求"图片重生成（如背景替换）"，实际做到了：

- **3 种背景风格可选** — 纯白底（Amazon 主图要求的标准白底）、生活场景（将产品放入真实使用环境）、渐变（白到灰的专业摄影风格）。商家可在抽屉图片 Tab 中一键切换风格后重新生成。合同未指定需要支持多种风格。
- **快速换背景预览** — 抽屉图片 Tab 中新增"快速预览"按钮，点击后直接用当前原图 URL 调用 AI 接口换背景，结果在弹窗中即时展示。商家无需走完整的"生成→保存→上传"流程，适合快速验证某种风格的效果后再决定是否正式生成。合同未提及此能力。
- **参考图支持** — 可提供一张参考图片的 URL，AI 会参照该图片的风格/背景来生成新图。例如提供一张竞品的场景图，AI 会生成类似场景但保留自己产品不变。合同未提及此能力。
- **图片合规校验（4 项自动检查）** — 每张 AI 生成的图片保存后，系统自动执行以下校验：
  - 最长边是否 ≥ 1000px（Amazon 要求，低于此尺寸无法启用缩放功能，影响转化）
  - 主图背景是否为纯白（通过采样图片四个角各 10×10 像素区域，计算平均亮度是否 ≥ 245/255，即接近 RGB 255,255,255）
  - 自动将 RGBA/PNG 图片转换为 RGB JPEG 格式（Amazon 推荐 JPEG）
  - 校验结果会在日志中输出，不合规项会标记警告
- 对应代码：`core/image_validation.py`（validate_image、check_white_background、ensure_jpeg）

### 5.2 文案处理方面的超额

合同只要求"标题/描述等字段生成或改写"，实际做到了：

- **从 4 类字段扩展到 7 类** — 合同提到的 4 类基础字段（标题、五点、描述、搜索词）全部支持。此外还额外新增了 3 类 Amazon 运营常用字段的 AI 生成：
  - 特殊功能（Special Features）— 2~5 个短语式产品亮点标签，如"BPA-Free Material""USB-C Fast Charging"
  - 目标受众（Target Audience Keywords）— 3~5 个受众标签，帮助 Amazon 匹配正确的买家群体
  - 主题关键词（Subject Keywords）— 补充标题中没有覆盖到的品类级别同义词，提升搜索覆盖率
- **AI 合规自动校验（4 个独立模块，参照 Amazon Seller Central 官方文档开发）：**
  - **标题校验**（`core/title_validation.py`）— 检测 Amazon 2025 年 1 月生效的新规：禁用字符（`! $ ? _ { } ^ ~ ¬ ¦`）自动删除、同一单词出现超过 2 次自动去重（介词/冠词/连词豁免，覆盖英/德/法/西/意/日等主要站点语言）、品牌名中的字符自动豁免
  - **五点校验**（`core/bullet_validation.py`）— 检测 Amazon 明确禁止的内容：价格信息（如"$19.99"）、运费信息（如"free shipping"）、退款承诺（如"money-back guarantee"）、公司信息（如"our company"）、emoji 和特殊符号。命中时标记警告
  - **搜索词校验**（`core/search_term_utils.py`）— 按站点区分字节限制（美国/欧洲 250 字节、日本 500 字节、印度 200 字节）；空格和标点不计入字节（符合 Amazon 官方规则，之前的算法多算了约 15%）；生成后自动移除标题和五点中已有的词（Amazon 明确说明重复词不提升排名，只浪费字节配额）；超过限制时自动按单词裁切而非暴力截断
  - **图片校验**（`core/image_validation.py`）— 见 5.1 节
- **全部对比视图** — 工具栏"全部对比"按钮，点击后弹窗以表格形式展示所有 SKU 的原始标题 vs AI 标题、原始图片 vs AI 图片、AI 五点、AI 搜索词的并排对比，方便商家一次性审阅全部 AI 生成结果，无需逐条打开抽屉。
- **system prompt 注入** — AI 文本生成时，系统自动注入一段 Amazon 专家角色的 system prompt（"You are an expert Amazon product listing copywriter and SEO specialist..."），确保所有生成内容的角色一致性和输出格式稳定性。OpenAI 协议走 system role message，Gemini 协议拼入 contents。自检和测试连接等非 Amazon 场景使用不同的 system prompt，互不干扰。

### 5.3 提交流程方面的超额

合同只要求"将 Excel 映射为接口请求并提交"，实际做到了：

- **Amazon VALIDATION_PREVIEW 预览门禁** — 正式提交前可先做一次 Amazon 官方的预览验证（SP-API `putListingsItem` 的 `VALIDATION_PREVIEW` 模式）。Amazon 会检查数据格式是否合规但不会实际创建商品。预览结果为 `VALID`（通过）或 `INVALID`（附带 issues 列表）。预览没通过的 SKU，系统会阻止进入正式提交流程。这大幅减少了正式提交后才发现被拒的情况。合同只要求"提交并获得回执"，预览门禁完全是额外做的。
- **本地字段校验（32 条规则）** — 提交前先在本地执行一轮校验，包括：必填字段完整性检查、标题长度和禁用字符、搜索词字节限制、图片地址可访问性、价格格式、UPC 格式、变体结构一致性（父子体 variation_theme 是否一致、子体是否有 parent_sku）等。校验不通过的 SKU 不会发送到 Amazon，避免浪费 API 配额。
- **5 层缺项诊断** — 点击"缺项诊断"后，系统对每个 SKU 执行 5 层检查并在弹窗中展示完整明细：
  1. **本地校验** — 上述 32 条规则的逐条结果，标注错误和警告
  2. **Amazon 预览** — 如果配置了 Amazon 账号，自动调用 VALIDATION_PREVIEW 获取 Amazon 侧的反馈，展示 issue code、severity、涉及字段名、中文修复建议
  3. **Listing 后台问题** — 如果该 SKU 已存在于 Amazon 后台，查询当前的 runtime issues
  4. **媒体连通性** — 逐张检查每张图片 URL 是否可正常访问（HTTP GET 验证）
  5. **缺失字段清单** — 汇总所有来源（本地+预览+后台）发现的缺失字段，附带字段名、中文标题、来源标签
  - 每个 issue 标注严重级别（红色=阻断必修、黄色=警告建议修），弹窗底部显示统计概览（错误数/警告数/缺失字段数/媒体失败数）

### 5.4 状态回执方面的超额

合同只要求"获得平台已接收/处理中等回执"，实际做到了：

- **ASIN 自动回查** — 提交返回 `ACCEPTED` 后，系统自动再查一次 `getListingsItem` API 尝试补获 ASIN。如果第一次未获取到，会间隔 0.5 秒后再查一次（最多 2 次）。获取到的 ASIN 自动回填到表格和 Excel 中。
- **提交路由识别** — 系统根据提交结果自动判断该 SKU 的上架路线：是全新创建商品，还是匹配到了已有 ASIN（跟卖）。路线信息展示在表格的"提交路由"列中。
- **最终 Listing 状态查询入口** — 在抽屉和结果区提供查看商品在 Amazon 后台最终状态的入口，可查看该 SKU 是否已进入 discoverable（可发现/可搜索）状态、当前后台 issues 数量、库存与报价状态等。
- **Amazon issue 中文说明与修复建议** — 系统内置了 20+ 个常见 Amazon issue code 的中文翻译和修复建议映射表（如 `MISSING_REQUIRED_ATTRIBUTE` → "缺少必填属性" → "请在 Excel 中补充该字段的值"），直接展示给商家，无需自行查阅 Amazon 开发文档。

### 5.5 提交日志方面的超额

合同只要求"输出提交日志/结果汇总"，实际做到了：

  - **最近记录** — 最近完成的预览/提交任务列表，每条显示时间、状态、成功/失败数，点击"查看详情"可看到每个 SKU 的逐条结果

- **Amazon 平台结果** — 是否接受请求、是否通过审核、是否上架（合同第一条第 3 款）
- **采集数据质量** — 数据的完整性、准确性、合法性（合同第五条第 1-2 款）
- **业务决策** — 品牌、资质、合规性、运营方向（合同第五条第 3 款）
- **商业结果** — 销量、转化、利润（合同第一条第 2 款）
- **验收标准** — 商品最终是否上架或产生销售，不作为验收依据（合同第七条第 3 款）

如因凭证失效、授权不足或数据缺失导致提交异常，建议甲方技术人员先参照第 11 节"技术交付验收确认清单"进行排查，开发者 2 可视情况协助定位。

## 9. 技术争议协商与 AI 辅助仲裁机制

### 9.1 协商优先原则

项目执行过程中如出现技术层面的分歧或争议，各方应本着合作精神优先协商解决。

### 9.2 AI 辅助仲裁机制

若经协商仍无法就技术事实达成一致，各方同意可采用以下方式辅助判定：

1. **发起方式：** 任何一方均可发起 AI 辅助仲裁请求。发起方负责组织会议并准备提交给 AI 的客观材料（代码、日志、接口返回、操作截图等），其他各方有权补充材料或提出补充问题。
2. **召集各方会议：** 由甲方（商家方）、开发者 1、开发者 2 共同参加。
3. **引入 AI 技术评审：** 将争议的技术事实及各方提供的客观证据，提交给当前业界公认的主流大语言模型（如 Claude 或 GPT）进行技术分析。
4. **AI 分析范围：** 仅限于技术事实判定（如"该错误是系统 bug 还是数据问题""该功能是否属于合同约定范围"等），不涉及商业决策或法律判断。
5. **参考性质：** AI 分析结论作为各方协商的**技术参考依据**，帮助厘清事实、促进达成共识，不具有法律裁决效力。

此机制旨在引入客观的技术分析能力，避免各方因信息不对称产生不必要的误解，同时确保任何一方在遇到争议时都有明确的启动路径。
