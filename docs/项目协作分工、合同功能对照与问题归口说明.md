| 阶段 | 内容 | 费用 |

合同第五条明确：

### 2.5 验收标准（合同第七条）

| 阶段 | 验收标准 |

## 4. 合同约定功能：逐条对照、是否完成、完成得怎么样

本节严格对照合同附件原文，逐条说明每项约定功能的交付情况。此部分为开发者 2 的合同义务，承担正常交付责任。

### 4.1 第一阶段：商品内容技术处理（合同附件第一节，费用 2,000 元）

合同附件第一节共列明 4 项内容，逐项对照如下：

**① 输入：甲方提供的 Excel 模板数据 + 图片素材（链接或文件）**

- 交付状态：**已交付 ✓**
- 系统支持导入 `.xlsx` 和 `.xls` 两种格式的 Excel 文件，导入后自动识别列名并映射到内部标准字段。图片素材支持公网 URL 链接和本地文件路径两种方式。
- 对应代码：`core/excel/processor.py`（ExcelProcessor.read_input）、`web/app.py`（/api/upload）

**② 图片处理：图片重生成（如背景替换）（按甲方确认风格/规则）**

- 交付状态：**已交付 ✓**
- 系统通过 AI 图片编辑接口对商品主图和副图进行背景替换处理。支持单条处理和批量处理。生成后的图片保存为本地文件并可上传至媒体存储（如 S3）获取公网地址。
- 对应代码：`core/ai_client.py`（ai_image_edit、ai_image_edit_url）、`stage1_pipeline.py`（_process_images）

**③ 文案处理：标题/描述等字段生成或改写（按甲方确认规则）**

- 交付状态：**已交付 ✓**
- 系统通过 AI 文本生成接口对商品标题、五点描述（Bullet Points）、商品描述（Description）、后台搜索关键词（Search Terms）进行生成或改写。支持单条和批量处理，生成后自动截断至 Amazon 官方字符/字节限制内。
- 对应代码：`core/ai_client.py`（ai_text）、`stage1_pipeline.py`（_generate_text_v2）、`core/prompts/amazon_prompts.py`（TITLE_PROMPT 等 13 个 prompt 模板）

**④ 输出：处理完成的 Excel 文件（字段与模板一致）**

- 交付状态：**已交付 ✓**
- AI 处理结果实时写回原始 Excel 文件（原始值保留，AI 结果写入新列用于前后对比）。同时支持两种格式导出：对比格式（原始 vs AI 并排）和 SP-API 格式（可直接用于提交）。
- 对应代码：`core/excel/processor.py`（write_comparison_output、write_output）、`web/app.py`（/api/export）

**结论：第一阶段合同约定 4 项功能已全部交付。**

### 4.2 第二阶段：自动化提交流程（合同附件第二节，费用 2,000 元）

合同附件第二节共列明 3 项内容，逐项对照如下：

**① 将输出 Excel 映射为平台接口请求并提交**

- 交付状态：**已交付 ✓**
- 系统读取第一阶段输出的 Excel 文件，自动将每行商品数据映射为符合 Amazon SP-API putListingsItem 接口要求的 attributes JSON 格式，然后通过 SP-API 逐条提交至 Amazon 美国站。映射逻辑覆盖产品身份、图片、价格、库存、物流、合规、变体等全部字段分组。
- 对应代码：`amazon/mapper.py`（FieldMapper.build_listing_attributes、build_put_body）、`amazon/listings.py`（ListingsAPI.put_listings_item）、`stage2_pipeline.py`（Stage2Pipeline.run）

**② 提交成功获得平台"已接收/处理中/草稿"等回执或状态**

- 交付状态：**已交付 ✓**
- 提交后系统接收并展示 Amazon 返回的状态信息，包括 `ACCEPTED`（已接受）、`INVALID`（验证不通过）、`ERROR`（接口错误）等状态，以及 `submissionId`（提交 ID）。如果 Amazon 分配了 ASIN，系统会自动回填。
- 对应代码：`amazon/listings.py`（_normalize_put_response、resolve_submission_asin）

**③ 输出：提交日志/结果汇总（含成功/失败原因提示，失败原因以平台回执为准）**

- 交付状态：**已交付 ✓**
- 提交结果逐 SKU 回写到 Excel 文件中，包含提交状态、提交 ID、ASIN、错误详情等字段。同时输出 JSON 格式的提交日志文件，包含完整的提交统计和逐条结果。
- 对应代码：`stage2_pipeline.py`（_generate_report）、`web/app.py`（任务中心相关接口）

**结论：第二阶段合同约定 3 项功能已全部交付。**

## 5. 超额完成：合同功能基础上主动增强的部分

以下内容的**基础功能属于合同约定**（已在第 4 节确认交付），但开发者 2 在实现过程中做到了显著超出合同基础描述的程度。超额部分并非合同要求，而是为了让商家实际使用时效果更好、出错更少而主动增强的。

### 5.1 图片处理方面的超额

合同只要求"图片重生成（如背景替换）"，实际做到了：

- **3 种背景风格可选** — 纯白底、生活场景、渐变，商家可按需选择，合同未指定需要多种风格
- **快速换背景预览** — 抽屉中点击"快速预览"可立即看到 AI 换背景效果，不需要走完整的批量流程
- **参考图支持** — 可上传一张参考图让 AI 按该风格生成，合同未提及此能力
- **图片合规校验** — 生成后自动检查：最长边是否 ≥ 1000px（Amazon 缩放要求）、主图背景是否为纯白（四角像素采样）、自动转换为 JPEG 格式（Amazon 推荐）

### 5.2 文案处理方面的超额

合同只要求"标题/描述等字段生成或改写"，实际做到了：

- **从 4 类扩展到 7 类字段** — 除合同约定的标题、五点、描述、搜索词外，还新增了特殊功能（Special Features）、目标受众（Target Audience）、主题关键词（Subject Keywords）的生成
- **AI 合规自动校验** — 参照 Amazon Seller Central 官方文档，生成后自动检测并修正：
  - 标题：禁用字符自动删除（`! $ ? _ { } ^` 等）、同一单词超过 2 次自动去重
  - 五点：检测是否包含价格、运费、退款承诺、公司信息、emoji（Amazon 禁止）
  - 搜索词：按站点字节限制裁切（美国 250 / 日本 500 / 印度 200）、空格和标点不计入字节（符合 Amazon 规则）、自动移除标题和五点中已有的词
  - 图片：生成后校验尺寸和白底
- **全部对比视图** — 工具栏"全部对比"按钮，弹窗展示所有 SKU 原始内容与 AI 生成内容的并排对比
- **system prompt 注入** — AI 生成时注入 Amazon 专家角色系统提示，提升输出格式稳定性和内容一致性

### 5.3 提交流程方面的超额

合同只要求"将 Excel 映射为接口请求并提交"，实际做到了：

- **Amazon VALIDATION_PREVIEW 预览门禁** — 正式提交前可先做一次预览验证，让 Amazon 检查数据格式是否合规，通过后再正式提交，大幅减少正式提交被拒的次数
- **本地字段校验** — 提交前先在本地检查必填字段是否完整、格式是否正确
- **5 层缺项诊断** — 诊断弹窗展示完整明细：本地校验结果 → Amazon 预览 issues → Listing 后台问题 → 每张图片的媒体连通性 → 缺失字段清单，每个 issue 标注严重级别和修复建议

### 5.4 状态回执方面的超额

合同只要求"获得平台已接收/处理中等回执"，实际做到了：

- **ASIN 自动回查** — 提交 ACCEPTED 后，系统自动再查一次 Listings API 补获 ASIN
- **提交路由识别** — 自动判断该 SKU 是新建商品还是匹配已有 ASIN
- **最终 Listing 状态查询入口** — 提供查看商品在 Amazon 后台最终状态（discoverable / suppressed 等）的入口

### 5.5 提交日志方面的超额

合同只要求"输出提交日志/结果汇总"，实际做到了：

- **任务中心 3 列** — 当前任务（实时进度）+ 最近记录（历史任务）+ 提交历史（累计提交统计），合同只要求输出日志文件
- **最近提交结果横幅** — 页面顶部展示最近一次提交的概要和快捷操作按钮
- **详情弹窗** — 点击任务可看到每个 SKU 的逐条结果明细
- **提交历史磁盘持久化** — 应用重启后历史记录不丢失
- **页面刷新自动恢复** — 刷新浏览器后自动恢复上次打开的文件，无需重新选择

> **超额完成部分的责任界定：**
>
> 以上第 5.1 至 5.5 节所列的增强内容，均超出合同附件的基础功能描述。开发者 2 对第 4 节所列的合同约定基础功能承担正常交付责任；超额增强部分属于主动提供的增值内容，其责任界定适用第 7 节条款。

## 6. 合同完全未要求、开发者 2 主动额外完成的工作

以下功能在合同正文及附件中**完全没有提及**。开发者 2 选择主动完成这些工作，是因为只交付合同最低要求的话，商家实际使用时会遇到很多本可以避免的问题，同时也为了降低开发者 1 采集侧的对接成本。

### 6.1 为提升商家使用体验额外完成的工作

**完整 Web 前端工作台**

- 合同仅要求数据处理与提交能力，未要求提供可视化操作界面
- 实际交付了包含 51 个 API 端点的完整 Web 工作台（5,372 行后端 + 11,000 行前端代码），商家无需使用命令行，全部操作可在浏览器中完成
- 包括：商品表格（搜索/筛选/分页/列设置）、右侧抽屉编辑（4 个 Tab）、任务中心、最近提交横幅、结果弹窗等

**18 章图文版商家使用文档**

- 合同未要求编写操作手册
- 实际编写了完整的《商家前端使用说明》（含术语表、3 套 SOP、FAQ、凭证获取指引），目的是让运营团队能自主上手，减少对开发者的依赖

**官方模板流程**

- 合同未提及类目推荐或模板生成
- 实际开发了完整的官方模板链路：输入链接/标题/关键词 → 推荐 Amazon product_type → 生成专属 Excel 模板（含中文说明行 + 稳定字段行）→ 上传采集结果 → 自动诊断 → 阻断项前置提示
- 解决了商家"不确定该选哪个 Amazon 类目"的实际痛点

**3 轮代码审阅与安全加固**

- 合同未要求代码审阅或安全加固
- 实际主动进行了 3 轮完整代码审阅，发现并修复 12 项安全/合规/体验问题（详见第 10 节），包括路径穿越防护、线程安全、凭证文件权限保护、XSS 防护等

**174 项自动化测试**

- 合同未要求编写测试
- 实际编写了 174 项自动化测试，覆盖全部核心模块（字段映射、校验、提交、AI 客户端、模板生成、图片处理等），确保系统稳定性和后续可维护性

### 6.2 为降低开发者 1 对接成本额外完成的工作

**灵活的列名自动识别**

- 合同未要求兼容多种列名格式
- 系统支持中英文多种列名别名映射（如"商品名称" / "item_name" / "标题" / "Seller SKU" / "卖家SKU" 等均可自动识别），采集侧无需严格对齐固定列名，大幅降低采集端适配工作量

**模板 Excel 含稳定字段行**

- 合同未要求生成可供采集插件使用的模板

## 9. 技术争议协商与 AI 辅助仲裁机制

### 9.1 协商优先原则

项目执行过程中如出现技术层面的分歧或争议，各方应本着合作精神优先协商解决。

### 9.2 AI 辅助仲裁机制

若经协商仍无法就技术事实达成一致，各方同意可采用以下方式辅助判定：

1. **发起方式：** 任何一方均可发起 AI 辅助仲裁请求。发起方负责组织会议并准备提交给 AI 的客观材料（代码、日志、接口返回、操作截图等），其他各方有权补充材料或提出补充问题。
2. **召集各方会议：** 由甲方（商家方）、开发者 1、开发者 2 共同参加。
3. **引入 AI 技术评审：** 将争议的技术事实及各方提供的客观证据，提交给当前业界公认的主流大语言模型（如 Claude 或 GPT）进行技术分析。
4. **AI 分析范围：** 仅限于技术事实判定（如"该错误是系统 bug 还是数据问题""该功能是否属于合同约定范围"等），不涉及商业决策或法律判断。
5. **参考性质：** AI 分析结论作为各方协商的**技术参考依据**，帮助厘清事实、促进达成共识，不具有法律裁决效力。

此机制旨在引入客观的技术分析能力，避免各方因信息不对称产生不必要的误解，同时确保任何一方在遇到争议时都有明确的启动路径。
