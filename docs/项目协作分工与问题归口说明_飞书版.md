# 亚马逊项目协作分工与问题归口说明

> 适用对象：甲方项目负责人、甲方技术人员、甲方运营人员、开发者 1、开发者 2
> 文档目的：明确项目执行过程中的协作分工、问题归口和沟通边界，减少理解偏差与责任混淆
> 说明：本文件用于项目执行与协作说明，不替代双方已签署合同；如与合同正文及附件《需求确认单/功能清单》的理解存在差异，以合同约定为准。

## 1. 为什么需要这份文档

本项目已经完成正式签约，但在实际执行过程中，往往会同时涉及：

- 数据采集与抓取
- 数据整理与模板映射
- AI 文案与 AI 图片处理
- 缺项诊断与 Amazon 预览
- SP-API 正式提交
- 账号权限、授权、联调与验收

如果不提前把分工说清楚，后续很容易出现下面几类问题：

- 采集问题和提交问题混在一起
- 商家运营把技术问题当成使用问题
- 商家技术把业务判断当成开发交付范围
- 本来属于采集链路的问题，被误认为是当前系统问题
- 本来属于平台审核或商家资质的问题，被误认为是开发未完成

因此，本文件的核心作用不是“强调谁不负责”，而是：

**把每类问题尽量快速归到正确的人和正确的模块，提升项目执行效率，减少不必要的反复沟通。**

## 2. 本文件和主使用说明的关系

当前项目建议保留两类文档：

### 2.1 主使用说明

主使用说明面向商家运营和商家技术，重点讲：

- 页面怎么用
- 按钮怎么点
- 预览怎么做
- 提交后怎么看结果
- 常见问题怎么排查

### 2.2 本协作文档

本协作文档只处理这些问题：

- 谁负责什么
- 哪类问题先找谁
- 哪些事项属于当前系统范围
- 哪些事项不属于当前系统范围
- 哪些需求如果新增，需要单独确认

简单理解：

- 主使用说明解决“怎么用”
- 本协作文档解决“找谁处理”

## 3. 当前项目参与角色说明

为便于协作，当前项目统一按以下角色划分：

### 3.1 开发者 1：浩浩

开发者 1 负责 **采集侧能力**，包括但不限于：

- 第三方店铺数据采集
- 爬虫抓取逻辑
- 采集站点适配
- 采集字段抓取与导出
- 采集图片链接抓取
- 采集结果结构输出

简化理解：

**凡是“商品数据是怎么抓下来、怎么导出来”的问题，优先看开发者 1。**

### 3.2 开发者 2：kk

开发者 2 负责 **导入后的亚马逊自动运营系统**，包括但不限于：

- 已有 Excel 数据的导入
- AI 文案处理
- AI 图片处理
- 官方模板流程
- 缺项诊断
- Amazon 预览
- SP-API 正式提交
- 任务中心、结果回显、状态展示
- 当前前端 Web 工作台
- 当前前端使用文档与提交流程说明

简化理解：

**凡是“数据导入系统之后，页面怎么处理、怎么诊断、怎么预览、怎么提交”的问题，优先看开发者 2。**

### 3.3 商家技术人员

商家技术人员负责：

- 提供 Seller ID、LWA Client ID、LWA Client Secret、Refresh Token
- 提供可用账号权限及必要授权
- 提供 AI 接口所需账号 / API Key（如需使用 AI）
- 配合环境、网络、联调与问题复现
- 配合排查账号权限、授权有效性、网络联通性等技术问题

简化理解：

**凡是“账号、授权、权限、联调、接口可用性”的问题，商家技术人员必须参与。**

### 3.4 商家运营人员

商家运营人员负责：

- 提供商品业务信息
- 确认标题、卖点、图片是否符合业务要求
- 确认品牌、类目、条码、资质等业务事项
- 确认是否正式提交
- 对 Amazon 后台审核、上架、可售、品牌归属等业务结果做判断

简化理解：

**凡是“这条商品是否适合卖、是否适合提、是否符合品牌与运营要求”的问题，属于商家运营判断范围。**

## 4. 当前项目的基础边界

### 4.1 当前系统负责到哪一步

当前系统负责到：

- Excel 数据导入
- 字段编辑与抽屉修正
- AI 文案与 AI 图片能力
- 缺项诊断
- Amazon 预览
- 正式提交到 Amazon SP-API
- 提交结果、任务记录、状态回显

如果一句话概括，就是：

**系统负责把“已经整理好的商品数据”尽量稳定地处理、校验、预览并提交到 Amazon。**

### 4.2 当前系统不负责哪些事项

下面这些事项，不属于开发者 2 当前系统的直接交付范围：

- 第三方店铺数据采集
- 爬虫抓取逻辑
- 采集站点页面适配
- 采集插件开发
- 采集源变化导致的字段缺失或错位
- 采集出来的图片链接失效
- 平台审核通过保证
- 品牌备案、类目资质、侵权、相似度、法务判断
- 销量结果、转化结果、运营结果保证
- 长期托管式运营服务

这里的表达不是为了回避配合，而是为了确保项目执行时，大家对模块边界有一致理解。

### 4.3 第三方平台结果边界

本项目涉及 Amazon 及其官方接口规则。

因此需要明确：

- Amazon 是否接受请求
- Amazon 是否通过审核
- Amazon 是否生成最终可售状态
- Amazon 是否因为品牌、条码、资质、类目等原因限制展示

这些结果首先属于 **第三方平台自身规则与处理结果**，不应简单等同于开发者行为。

## 5. 问题归口总原则

建议项目执行中统一使用下面这条归口原则：

**采集问题找开发者 1（浩浩），导入后的处理 / 诊断 / 预览 / 提交问题找开发者 2（kk），账号权限与授权问题由商家技术先配合排查，品牌资质与运营决策问题由商家侧判断。**

这句话建议直接写进：

- 飞书项目群公告
- 对接说明
- 验收沟通记录
- 日常问题归口文档

这样后续大家看到问题时，能先按模块归口，而不是先从“谁背锅”开始。

## 6. 常见问题归口表

下面是执行中最常见的问题类型，以及建议的第一归口人。

### 6.1 采集与源数据问题

#### 情况 A：Excel 里没有 SKU / 标题 / 主图 / 价格这些基础列

建议归口：**开发者 1（浩浩）优先处理**

原因：

这属于采集输出结构问题，问题发生在“进入当前系统之前”。

#### 情况 B：采集出来的列名和系统模板对不上

建议归口：**开发者 1（浩浩）优先处理，开发者 2（kk）可配合确认模板字段要求**

原因：

这是上下游字段约定问题，核心仍属于采集输出侧。

#### 情况 C：采集图片链接打不开 / 链接失效 / 不是图片直链

建议归口：**开发者 1（浩浩）优先处理**

原因：

图片来源和采集输出链路属于采集模块责任范围。

#### 情况 D：采集站点改版后字段变少、字段错位、变体关系缺失

建议归口：**开发者 1（浩浩）优先处理**

原因：

这是典型的采集站点适配问题。

### 6.2 当前系统处理问题

#### 情况 E：导入后页面显示异常、字段映射异常、抽屉展示异常

建议归口：**开发者 2（kk）优先处理**

原因：

问题发生在数据已经进入系统之后。

#### 情况 F：AI 文案生成结果明显不合理

建议归口：**开发者 2（kk）优先处理，商家运营参与判断**

原因：

技术上属于当前系统；业务上需要运营判断是否符合商品定位。

#### 情况 G：AI 图片换背景效果不好

建议归口：**开发者 2（kk）优先处理，商家运营参与确认风格要求**

#### 情况 H：缺项诊断结果看不懂 / 诊断异常

建议归口：**开发者 2（kk）优先处理**

#### 情况 I：Amazon 预览失败

建议归口：**开发者 2（kk）优先协助排查，商家技术与商家运营视问题类型配合**

因为预览失败可能来自：

- 字段格式问题
- 图片问题
- 品牌 / 类目 / 条码问题
- 账号权限问题

所以归口虽然先到开发者 2，但不一定全部由开发者 2 单独解决。

#### 情况 J：正式提交失败或提交状态异常

建议归口：**开发者 2（kk）优先协助排查，商家技术必须配合**

### 6.3 商家技术问题

#### 情况 K：账号测试失败

建议归口：**商家技术人员先配合排查，开发者 2（kk）协助**

优先检查：

- Seller ID 是否正确
- LWA Client ID / Secret 是否填反
- Refresh Token 是否有效
- 账号与授权是否匹配

#### 情况 L：AI 能用，但 Amazon 提交不通

建议归口：**商家技术 + 开发者 2 联合排查**

原因：

AI 配置和亚马逊账号配置是两套系统，不能互相替代判断。

### 6.4 商家运营判断问题

#### 情况 M：品牌、类目、资质、条码是否适合提交

建议归口：**商家运营 / 商家技术内部先判断**

#### 情况 N：Amazon 最终审核结果、是否继续卖、是否更换文案方向

建议归口：**商家运营负责最终业务判断**

开发者可以协助解释系统状态，但不替代业务决策。

## 7. 非当前交付范围说明

为了减少执行过程中的误解，这里把常见“容易被误认为默认包含”的事项单独列出来。

以下事项，如未在合同正文或附件中另行书面确认，一般不应默认视为开发者 2 当前交付义务：

- 新增采集源接入
- 新增爬虫逻辑
- 新增第三方站点适配
- 长期托管式维护
- 无限次规则变化跟进
- 额外的品牌备案与资质协助
- 平台审核通过结果保证
- 销量与商业结果保证
- 超出既有范围的新页面 / 新系统 / 新流程开发

如果后续确有需要，建议按“新增需求 / 变更需求”单独确认。

## 8. 对外沟通建议用语

为了既不显得强硬，又能把边界讲清楚，建议在飞书群或沟通中使用下面这种口径。

### 8.1 归口说明口径

可以这样说：

“为方便排查，建议先按模块归口：采集链路相关问题先看开发者 1，导入后的处理、诊断、预览和提交相关问题先看开发者 2；账号权限和授权问题请商家技术协助确认，品牌和运营判断由商家侧决定。”

### 8.2 对客户的委婉表达方式

可以这样说：

“为了更高效地推进，我们建议把采集侧问题和导入后的运营系统问题分开处理。这样每类问题都能更快找到对应负责人，避免不同模块的问题混在一起影响处理效率。”

### 8.3 对内的明确口径

对内建议更直接一些：

“采集问题归浩浩，导入后处理和提交问题归 kk，账号权限商家技术先确认，业务判断商家运营负责。”

## 9. 技术交付前的建议确认清单

如果商家技术人员准备接手使用或联调，建议先确认这份清单：

1. Seller ID 已确认
2. LWA Client ID 已确认
3. LWA Client Secret 已确认
4. Refresh Token 已确认
5. 默认账号已设定
6. 测试按钮已通过
7. AI Base / API Key 已确认（如需使用 AI）
8. 至少已完成一次单品预览
9. 至少已完成一次单品提交演练
10. 至少已完成一次变体家族演练

只要这 10 条中前 7 条都没确认，后面正式跑流程时就很容易反复卡住。

## 10. 项目执行中的建议协作方式

为了让合作过程更顺，建议保持下面这种协作方式：

### 10.1 出问题先带 4 个信息

无论找谁，建议先附上：

- 当前文件名
- 具体 SKU
- 页面截图
- 发生在哪一步（导入 / AI / 诊断 / 预览 / 提交）

### 10.2 技术问题尽量先复现再归口

不要只说“不能用”，而是尽量描述成：

- 哪个文件
- 哪个 SKU
- 点了哪个按钮
- 看到了什么结果
- 预期应该是什么

### 10.3 采集和系统联调时，先固定一版字段结构

建议先让采集侧和当前系统对齐：

- 基础字段
- 价格字段
- 图片字段
- 变体字段
- 模板字段

字段口径固定后，后面会省掉很多反复调试。
